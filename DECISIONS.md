# DECISIONS
This is exhaustive, for principles it's based off, see my hand-written convictions.md.

## Scope & priority
**Target:** aim for everything, but only advance to the next tier after all tests pass *and* an adversarial architecture review (human-judged) passes. Quality gate per tier, not raw coverage.

Build in this order; a few features done cleanly beats many half-done.
1. **P0** ‚Äî config load, `/health`, prefix proxying (longest match, 404), method filter (405), schema-general.
2. **P1** ‚Äî rate limiting (global + per-route) **first** (namesake + concurrency proof), then `strip_prefix`, then timeouts (global + per-route ‚Üí 504).
3. **P2** ‚Äî api_key auth **first** (cheapest stage, proves the architecture), then retry/backoff, circuit breaker, load balancing, health checks.
4. **P3** ‚Äî request/response transforms (headers, body mapping, envelope).

**Grade-optimal floor:** P0 + rate limiting (both strategies, concurrency-proven) + one clean resilience feature + auth + flawless DECISIONS/README. Beyond the floor is bonus; stub cleanly, never half-wire. Per-tier review gate must not burn the clock.

## Architecture
- **Pluggable pipeline (extensibility ‚Äî the 60%-of-grade decision):** the pipeline is an ordered list of composable **stages**, each a `Stage` trait (`async fn apply(&mut ctx) -> Flow { Continue | ShortCircuit(Response) }`), assembled per-route from config. New config feature = config struct + one stage + register; no core-loop change. This is the criteria.md "extend in an afternoon" litmus test.
- **Module map:** `config/` (parse+validate), `pipeline/` (`Stage` trait + one file per stage), `upstream/` (client, target pool, health), `server` (hyper wiring, `RequestCtx`, error‚Üístatus, access log), `mock_upstream` (tests).
- Pipeline order: `match route ‚Üí method ‚Üí auth ‚Üí rate limit ‚Üí request transform ‚Üí circuit-breaker gate ‚Üí select target ‚Üí timeout+retry around upstream ‚Üí response transform ‚Üí return`. Fast rejections (404/405/401/429) before any upstream work.
- All state in-memory, single instance (spec blesses this). Rate-limit counters, breaker state, LB cursors, health status are shared mutable state ‚Äî race-free via **sharded maps / atomic per-entry counters** (no map-level lock serializing throughput).
- **HTTP stack:** `hyper` (+ `hyper-util`) used **only as a low-level HTTP transport** ‚Äî Rust std has no HTTP; closest legit analogue to std-lib HTTP. All routing/matching/proxy logic hand-built; no gateway/reverse-proxy crate.
- **Proxy hygiene:** strip hop-by-hop headers both directions (`Connection`/`Transfer-Encoding`/`Keep-Alive`/`Upgrade`/`Proxy-*`), rewrite `Host` to upstream, recompute `Content-Length` on body change.
- Parse all durations and enums (`strategy`, `backoff`, `balance`, `per`, `auth.type`) at load; **cross-field validation** (weighted RR needs weights; `url` xor `targets`; `health_check` needs `path`); reject unknown values with non-zero exit.
- **Observability:** one structured access-log line per request (route, decision/status, latency, target) + error logging.
- **Body limits:** cap max body size + read timeout even with transforms stubbed; oversize ‚Üí 413.

### Tier-1 as-built (where the impl concretizes the plan)
- **Module layout:** `pipeline/` and `upstream/` are directories (the extensibility surface); `config`/`server`/`health`/`router`/`error` stay single-file modules (churn not worth it yet).
- **Route match + method:** route match runs *before* the chain (it selects which chain to run); `MethodStage` is the first ‚Äî and, in P0, only ‚Äî registered stage.
- **Deferred to when it has teeth:** body-size cap + 413 needs request-body **buffering**, which is coupled to P1 retry and P3 transform buffering ‚Äî landing it there avoids committing a buffering design twice. `Content-Length` recompute likewise lands with body transforms (P3); P0 relays the body as a stream so hyper reframes it.

## Resolved ambiguities (spec silent ‚Üí our call)
- **Route match:** **segment-boundary** longest prefix (`/api/users` matches `/api/users` + `/api/users/‚Ä¶`, not `/api/usersXYZ`). Path matched first; method mismatch on the chosen route ‚Üí 405 (no fallback to a shorter route).
- **Port:** bind `gateway.port` don't hardcode this because it's something a future dev may wish to configure
- **Client IP (`per: ip`):** when ratelimiting on IP, don't trust X-Forwarded-For header, use socket peer
- **Rate-limit bucket identity per route**
- **Rate limit override:** route `rate_limit` fully replaces global (no merge). Over limit ‚Üí **429 + `Retry-After`** (int seconds to capacity). `sliding_window` = sliding-counter (weighted current+previous window, O(1)/key). Sharded map + idle-key eviction (unbounded per-IP growth).
- **Timeout placement/scope:** accept `timeout` under `upstream` *and* at route level; route/upstream value beats global. Timeout is **per attempt**; separate overall wall-clock budget caps retries+backoff. (Note: in the example config, both `/api/orders` and `/api/products` place `timeout` under `upstream`.)
- **Retry:** `attempts` = total tries. Retry on listed statuses **and** transport errors/timeouts. Honor config even for non-idempotent methods (at-least-once risk accepted). On retry, advance to **next healthy target**. A fully-failed retried request = **one** breaker failure (per-request, not per-attempt); retry sits inside the breaker gate.
- **Circuit breaker:** **per-target** (per-upstream), single-probe half-open; open ‚Üí 503 envelope `{ "error": "service_unavailable", "retry_after": <s> }`. Counts 5xx/timeouts/connection errors. Single-`url` upstream = one breaker.
- **Auth:** missing *or* invalid key ‚Üí 401 (no 403 distinction); constant-time compare. Runs **before** rate limiting (protects key compare; `per` is ip/global so no identity needed to bucket). Tradeoff: rate-limit-first would shed bad-key floods cheaper ‚Äî accepted.
- **Health checks:** active probe; mark unhealthy after `unhealthy_threshold` consecutive failures; recover on first success; all targets unhealthy ‚Üí 503. **Passive ejection:** live-traffic failures feed the per-target breaker so a dead target is ejected before its next probe. `health_check` applies to single-`url` upstreams too.
- **Load balancing:** `round_robin` ignores `weight`; `weighted_round_robin` = smooth weighted RR; atomic cursor; skip unhealthy/Open targets.
- **Request/response body transform:** only for JSON bodies; non-JSON request body passes through unchanged. Response envelope: non-JSON/empty upstream body embedded as a JSON string under `$body`; envelope applies only to real upstream responses, **not** gateway-generated errors.
- **Dynamic values:** `$request_time`/`$response_time` = RFC-3339 UTC, computed once per request/response (header + body-mapping uses agree). `$route_path` = matched route path.
- **405** carries an `Allow` header; **413** for oversize request bodies.

## Error mapping
404 unmatched ¬∑ 405 bad method (+`Allow`) ¬∑ 401 auth ¬∑ 429 rate limit (+`Retry-After`) ¬∑ 413 oversize body ¬∑ 503 circuit open / all targets unhealthy ¬∑ 504 upstream timeout ¬∑ 502 upstream connection error. No panic leaks to clients.

## What's next / partial
- **P0 complete** (as-built above): config load + cross-field validation, `/health`, longest-prefix segment-boundary proxying + 404, method filter (405 + `Allow`), schema-general, pluggable `Stage` pipeline, access logging.
- **P1 complete:** rate limiting (fixed + sliding window, `per: ip`/`global`, sharded map + atomic-per-bucket check-and-increment proven exact under 50 concurrent requests, background idle-key eviction, 429 + `Retry-After`); `strip_prefix` (segment-tail rewrite, query preserved); per-attempt timeouts (route ‚Üí upstream ‚Üí global precedence, 504). Rate limiting is a `RateLimitStage` pushed by `pipeline::assemble` only for routes with an effective limit.
- **P2 complete:** api_key auth (`AuthStage`, before rate-limit, constant-time compare, 401); retry+backoff (fixed/exponential, on statuses + transport/timeout, per-attempt timeout + overall budget, fails over to next eligible target); per-target circuit breaker (single-probe half-open, 503 `service_unavailable`+`retry_after` envelope); load balancing (round-robin + smooth weighted RR, skips unhealthy/Open targets); active health checks (eject after threshold, recover on first success) with passive ejection between probes (live-traffic failures feed both the breaker and a per-target passive-health counter, so ejection works even with no breaker configured). All P2 target state (breakers, health flags, LB cursors) lives in one per-route `UpstreamRegistry` (indexed like the rate limiter), and the request body cap + 413 landed here (retry buffers the body).
- **P3 complete:** request transform ‚Äî header add/remove (`RequestTransformStage`, `$request_time`/`$literal` values) + JSON body mapping (dot-path `dest <- source`, `$literal`/`$request_time` sources, non-JSON passthrough); response transform ‚Äî header add/remove + body envelope (`$body`/`$response_time`/`$route_path`), non-JSON/empty upstream body embedded as a JSON string, applied only to genuine upstream responses (gateway errors never enveloped). `Content-Length` recompute for transformed bodies now falls out for free (both directions rebuild a `Full` body from known-length bytes; stale framing headers stripped so hyper reframes).

### Tier-2 as-built (where the impl concretizes the plan)
- **Unified upstream runtime.** The three reserved seam registries (breaker/balancer/health) collapsed into **one** `upstream::UpstreamRegistry` (per-route `RouteUpstream` = `Vec<Arc<TargetRuntime>>` + `Balancer` + optional health probe). A single-`url` upstream is modelled as a **one-target pool**, so breakers and health apply to it uniformly. Rationale: LB, breaker, and health all key on the *same* per-target identity and are consulted together during selection; splitting them across three registries would have forced three parallel lookups and a shared index invariant. Selection is one scan: skip health-unhealthy, skip breaker-Open, take the first eligible (a granted half-open probe counts).
- **Two 503 flavours, one selection.** If selection finds nothing eligible: any target held out by an **Open breaker** ‚Üí `CircuitOpen{retry_after}` (soonest cooldown remaining) = the spec's envelope; otherwise all exclusions were **health** ejections ‚Üí `AllTargetsUnhealthy`.
- **Breaker accounting reconciliation (per-request vs per-attempt).** DECISIONS said "a fully-failed request = one breaker failure." With per-target failover that's ambiguous, so the resolution is: record outcomes **per target, at most one failure per target per request** (`failed_targets` set). Single-`url` retried 3√ó and failing = one failure (matches the decision); with failover each *distinct* failed target gets one. A `<500` response records success (target is alive) even if its status is in `retry.on`.
- **Retry ‚üÇ breaker-failure classification.** Whether to *retry* (status ‚àà `retry.on`, timeout, or transport) is a **separate** decision from whether it's a *breaker failure* (5xx / timeout / transport). A 500 not in `retry.on` is relayed to the client but still counts against the breaker.
- **Body cap = 2 MiB (`MAX_BODY_BYTES`), enforced while reading** (oversize ‚Üí 413 without buffering it all). **Overall retry budget** = `attempts * per_attempt_timeout + Œ£ backoffs` when a per-attempt timeout is set; with no timeout the finite attempt count + backoff sum is the bound.
- **Balancer default.** Multiple `targets` with no explicit `balance` ‚Üí round-robin (sensible default); a lone target ‚Üí no balancing (`Single`). `circuit_breaker.threshold: 0` is clamped to 1 (validation doesn't forbid it) so a breaker is always meaningful.
- **Extensibility surfaces ‚Äî bounding the "add one stage" claim (honest scope).** The `Stage`/`ResponseStage` chains are the extension point for **request- and response-phase policies** (method, auth, rate limit, transforms) ‚Äî those genuinely are "add a config struct + one stage + register." **Upstream-call concerns** (retry, circuit breaker, load balancing, health) are *not* stages and deliberately can't be: they loop, select among targets, and wrap the terminal call. Their extension surface is the `upstream` module ‚Äî a new balancer strategy or breaker policy is a clean enum-variant + one match arm (an afternoon job), but a wholly new upstream-call concern means composing into `upstream::proxy`'s retry loop, not dropping in a stage. So the litmus test holds per-surface; it is not one universal mechanism, and we don't claim it is.
- **Unconditional request-body buffering (accepted tradeoff).** `upstream::proxy` buffers every request body once (retry needs a re-sendable body; P3 request-body mapping reuses the same buffer). Consequence vs. P0's streaming: even a route with neither retry nor transform now buffers up to `MAX_BODY_BYTES` and returns 413 above it, and a slow body is bounded by a 30s read timeout (‚Üí408). Accepted as a uniform DoS posture (bounded memory + no slow-loris) that also keeps one body-handling path; the alternative (stream when no retry/transform) was judged not worth the branching.

### Tier-3 as-built (where the impl concretizes the plan)
- **Transforms extend the *same* pluggable pipeline ‚Äî in both directions.** A P1 review had already added a symmetric response phase (a `ResponseStage` trait + `assemble_response` + `state.response_stages`, mirroring the request `Stage`/`assemble`). P3 slots straight into both seams: request headers ‚Üí `RequestTransformStage` (a `Stage`), response headers+envelope ‚Üí `ResponseTransformStage` (a `ResponseStage`). Neither `handle`'s request loop nor its response loop changed ‚Äî this is the concrete payoff of the "extend a config feature in an afternoon" contract, now demonstrated on the response side too.
- **Where each transform runs (and why it isn't all one stage).** Request *header* rewrite is a pre-upstream `Stage` (operates on the request head). Request *body* mapping is **not** a stage: it runs at the body-buffer boundary in `upstream::proxy`, because the retry loop already buffers the body once ‚Äî transforming it there avoids buffering `Request<Incoming>` a second time (you can't swap an `Incoming` body for buffered bytes inside a `&mut RequestCtx`). Both read the **one** `RequestCtx.request_time`, so a `$request_time` header and a `$request_time` body field are identical to the second. Tradeoff: request-transform logic lives in two files; accepted, and it matches the pre-committed "body work lands where the body is buffered" note from Tier-1.
- **Response envelope only touches real upstream responses.** `run_response_stages` runs on the `Ok(resp)` arm of `upstream::proxy` only; every gateway-generated error (`404/401/429/503/504/413‚Ä¶`) takes the `Err` arm and skips the phase, so an envelope never wraps a `{"error":‚Ä¶}` body (spec/DECISIONS requirement). Enveloping buffers the response body (required to restructure it); header-only transforms leave the body **streamed** (no needless buffering of large responses). Non-JSON/empty upstream bodies embed as a JSON string under `$body` rather than 500 ‚Äî enveloping is presentation and must not fail an otherwise-good response.
- **Hand-rolled RFC-3339 UTC (no date crate).** `$request_time`/`$response_time` are formatted by a ~15-line civil-from-days routine (`pipeline::transform`), keeping the "build the gateway from scratch" boundary crisp. Unit-tested against known epochs incl. a leap day.
- **`$literal:` escape hatch + missing sources.** A value/source of `$literal:<v>` is taken verbatim (lets a config emit a value that starts with `$`). A body-mapping source dot-path that's absent in the incoming body is **skipped** (not written as null), so the mapped shape only carries fields that actually resolved.

## Process notes (tier gates & how the build actually went)
- **P1 built by two parallel agent teams** on disjoint files (proxy features vs. rate limiting), with shared error variants pre-added so there were no merge conflicts; compilation deferred to tier end per the speed tradeoff.
- **P2 built by two parallel agent teams** on disjoint files (auth stage vs. the resilience/upstream layer), with the shared 401/503/503/413 error variants pre-added first so there were no merge conflicts ‚Äî same play as P1. Compilation deferred to tier end; the single integration fix was the P0.5 alternate-config test, whose auth-carrying route now enforces the key (200 with it, 401 without) instead of proxying unauthenticated. Full suite green (92 tests) before the tier-2 review gate.
- **P2 tier-2 review (adversarial, 3 parallel reviewers: correctness / architecture / spec-compliance; human-judged).** Verdict: pass with iteration. Human accepted fixing the findings that matter and deferring polish. **Fixed:** (1) circuit breaker was resetting its rolling failure count on every non-5xx, so a *partially*-degrading upstream (503,200,503‚Ä¶) never tripped ‚Äî now failures age out by time and only a half-open trial success closes it (+ interleaved-failure regression test); (2) passive ejection depended on a `circuit_breaker` being configured, leaving a `health_check`-only route (the spec's `/api/products`) blind for a full probe interval ‚Äî live failures now feed a per-target passive-health streak too (eject after threshold, active-prober recovery), covered by a no-breaker integration test; (3) no inbound read timeout ‚Äî added a 30s body-read timeout (‚Üí408) and connection `header_read_timeout` (slow-loris guard). **Documented/deferred (human's call):** the "add one stage" scope note + unconditional-buffering tradeoff above; access-log chosen-target, `Connection`-token stripping, breaker/balancer lock-poison recovery, and the remaining acceptance-test gaps (exponential backoff timing, 504-vs-502 exhaustion, POST retry) left as noted polish.
- **P3 built by three parallel agent teams** on disjoint new files (request-transform module, response-transform module, integration tests + mock extension), with the shared `pipeline::transform` util (timestamp, value resolution, JSON dot-path) and the conflict-sensitive wiring (`RequestCtx.request_time`, the two `assemble*` registrations, the `upstream::proxy` body-mapping hook) held by the orchestrator. Compilation deferred to tier end; the single integration fix was the P0.5 alternate-config test, whose response-transform route now returns the **live** envelope (proving P3 wires on a brand-new config, not just the example). Full suite green (119 tests) before the tier-3 review gate. Concurrency note: a P1-review agent landed the symmetric response-phase seam mid-tier; P3 adopted it (`ResponseTransformStage: ResponseStage`) rather than the originally-planned free function ‚Äî the cleaner design won.
- **Mid-P1 architecture refactor (kept, decided by human):** during P1 an agent refactored the P0 pipeline from ordered free-functions into the promised pluggable `Stage`-trait module (`pipeline/`, `upstream/`) and folded it in as "P0 as-built." This overshot its scope and briefly regressed `strip_prefix`/timeouts (re-applied) and required one corrected rate-limit test. **Decision:** keep the `Stage`-trait design ‚Äî it is exactly the extensibility contract this doc promises (criteria.md's "extend in an afternoon") and rate limiting already builds on it; roll-back would have reopened the docs‚Üîcode gap. **Decision:** keep the existing commits and add this honest note rather than rewrite history (nothing was pushed).
