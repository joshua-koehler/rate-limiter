GatewayKit — SWE Take-Home
Welcome
Thanks for taking the time to complete this exercise. We know your time is valuable.
A Note on Scope
This exercise is intentionally larger than what can be completed in 2 hours. We designed it
this way on purpose. We expect you to use AI coding tools (Cursor, Claude Code, etc…), and
we want to see how you orchestrate your AI IDE to make strategic progress on an ambitious
problem under time pressure.
No one will finish everything. What separates strong candidates is not how much they
complete, but the quality of what they build, the order in which they tackle things, and how
clearly they reason about what they left out. A well-architected gateway that handles three
features cleanly will always beat a brittle gateway that half-implements six.
You have 2 hours from receiving these requirements to submit. After submission, we'll schedule
a 30-minute code walkthrough where you'll present your solution and discuss your decisions.
You should be prepared to explain any code in your submission — even code generated by AI
tools.
The Problem
Your team has been asked to build GatewayKit, a lightweight, config-driven API gateway. It sits
between clients and a set of upstream services, handling routing, rate limiting, request/response
transformation, and resilience.
Think of it as a simplified version of tools like Kong, Envoy, or AWS API Gateway — built from
scratch.
How This Works
We're providing you with a complete gateway configuration file (gateway.yaml, included
below). This config file is the spec. Your job is to build a gateway that reads this file and
behaves accordingly.
None
The config is self-documenting — field names describe what they do. Implement as much of the
configuration as you can within the time constraint. You will not finish all of it, and that's
expected. We want to see how you prioritize, how you architect for extensibility, and how you
handle the features you do implement.
We will evaluate your submission by running your gateway against the provided config, then
running it again with a different config file that uses the same schema but different values and
routes. Your gateway should work with any valid config that follows this schema — not just the
example provided.

Core Requirements
These are non-negotiable baseline expectations:
1. The gateway must start and listen on port 8080, reading its configuration from a
YAML file path passed as a command-line argument or environment variable.
2. GET /health must always return 200 OK with { "status": "healthy",
"uptime_seconds": <int> } regardless of config.
3. Basic proxying must work. Requests matching a configured route should be forwarded
to the upstream and the response returned to the client. Unmatched routes should return
404.
4. Method filtering must work. A POST to a route configured for ["GET"] only should
return 405 Method Not Allowed.
5. Your gateway must work with any valid config following this schema, not just the
example config provided. We will test with a different config file.
What to Submit
1. A Git repository (GitHub link, GitLab link, or zip with .git history intact). Commit
history matters — we want to see how you worked, not just the final result.
2. A DECISIONS.md file in the repo root documenting:
- How you prioritized which config features to implement
- Architectural choices and trade-offs (e.g., how you structured the proxy pipeline,
how you approached extensibility)
- What you'd build next with more time
- Any features you partially implemented and their current state
- How you used AI tools (optional but encouraged — we're genuinely curious)
3. A working test suite runnable with a single command. Include a mock upstream server
or test harness so tests are self-contained.
4. A README.md with:
- How to set up and run the gateway
- How to run the tests
- Which config features are implemented (a checklist is fine)
- Any prerequisites or dependencies

Constraints
- Language/framework: Your choice — use what you're strongest in. You may use your
language's standard library HTTP server/client and a YAML parsing library. Do not use
existing API gateway, reverse proxy, or HTTP proxy frameworks/libraries. Building
the proxy logic is the point.
- Data storage: In-memory is perfectly fine. No database required.
- Mock upstreams: Include a simple upstream server in your repo for testing. It doesn't
need to be fancy — a few endpoints that return canned responses (and maybe one
that's intentionally slow or flaky) will do.
